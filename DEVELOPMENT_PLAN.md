# 《天元迷局》开发设计文档

> 项目类型：2D 策略 Roguelike / 变体五子棋  
> 推荐技术栈：Godot 4.x + GDScript  
> 目标平台：PC 优先，后续扩展至移动端与 Web 试玩  
> 文档用途：指导后续原型、MVP、正式版本开发与团队协作

---

## 1. 项目愿景

《天元迷局》是一款以五子棋为核心规则的 2D 策略 Roguelike 游戏。玩家扮演棋术师，在随机生成的棋盘关卡中挑战不同敌人，通过技能、地形、棋子强化、路线选择和奖励构筑，形成不同战斗流派，并最终击败 Boss。

本项目的设计目标是：

- 保留五子棋“连成五子即可获胜”的直觉核心。
- 引入 Roguelike 的随机性、成长性和重复游玩价值。
- 通过技能、地形和特殊棋子制造战术变化。
- 让敌人拥有不同棋风，而不是只提高 AI 难度。
- 让每一次通关过程都能形成不同的构筑故事。

一句话概念：

> 用 Roguelike 的构筑体验重塑五子棋，让每一局棋都是一次冒险。

## 2. 技术栈选择

### 2.1 首选技术栈

| 模块 | 选择 |
| --- | --- |
| 游戏引擎 | Godot 4.x |
| 主要语言 | GDScript |
| 目标平台 | Windows / macOS / Linux |
| 后续平台 | Android / iOS / Web |
| 渲染方向 | 2D |
| 数据配置 | JSON + Godot Resource |
| 存档 | 本地 JSON / ConfigFile |
| 版本管理 | Git |
| 项目管理 | GitHub Issues / Notion / Trello |

### 2.2 选择 Godot 的原因

Godot 适合本项目的主要原因：

1. **2D 表现能力强**：棋盘、棋子、地形、技能范围、UI 面板都属于典型 2D 游戏元素。
2. **原型迭代快**：GDScript 语法轻量，适合玩法快速验证。
3. **项目体量适配**：本项目不需要重型 3D 引擎，Godot 更轻便。
4. **授权友好**：Godot 免费开源，适合独立游戏与小团队。
5. **多平台导出**：PC、移动端和 Web 都可覆盖。
6. **节点系统清晰**：适合拆分棋盘、UI、效果、技能与 Roguelike 地图。

### 2.3 为什么暂不首选 Unity

Unity 依旧是可行方案，尤其适合已有 Unity 经验的团队。但对本项目而言：

- 引擎相对更重。
- 授权、版本和商业策略需要额外关注。
- 2D 棋盘策略游戏不需要 Unity 的完整 3D 能力。
- 小团队原型阶段使用 Godot 更直接。

### 2.4 为什么暂不首选 Web 技术栈

Phaser + TypeScript 适合快速做网页 Demo，但如果目标是 PC / Steam 单机游戏，Godot 在打包、资源管理、音频、输入、场景组织和游戏循环方面更完整。

## 3. 游戏核心规则

### 3.1 基础胜利条件

默认规则为：

> 玩家或敌人在横、竖、斜任意方向连续形成五颗己方棋子时获胜。

为了适配 Roguelike，部分特殊关卡可以扩展胜利条件：

- 限定回合内连五。
- 在指定地形上完成连五。
- 连成六子才能击败 Boss。
- 同时形成两个三连阵型。
- 保护核心棋子若干回合。
- 封锁敌方特殊棋子。

MVP 阶段只实现标准连五胜利。

### 3.2 回合结构

推荐回合流程：

```text
玩家回合开始
  -> 触发回合开始效果
  -> 获得基础能量
  -> 玩家选择落子或使用技能
  -> 结算行动效果
  -> 结算地形与状态
  -> 检查胜负
敌人回合开始
  -> 触发敌方回合开始效果
  -> AI 选择行动
  -> 结算行动效果
  -> 结算地形与状态
  -> 检查胜负
```

MVP 阶段建议简化为：

- 每回合必须落一颗棋。
- 技能可以在落子前或落子后使用。
- 每回合基础获得 1 点能量。

### 3.3 能量系统

能量是限制技能频率的核心资源。

建议基础规则：

- 每回合开始获得 1 点能量。
- 初始能量上限为 6。
- 构筑奖励可将能量上限提升到 8 或 10。
- 在灵脉格落子可额外获得 1 点能量。
- 形成三连、四连或成功防守可触发额外回能。

技能费用建议：

| 技能类型 | 推荐费用 |
| --- | ---: |
| 小型提示 / 侦察 | 1 |
| 移动己方棋子 | 2 |
| 移除地形 | 2 |
| 放置障碍 | 2 |
| 封锁敌方落点 | 3 |
| 生成临时棋 | 3 |
| 传送 / 同调 | 4 |
| 大范围地形改造 | 5 |
| 传说级强技能 | 6+ |

## 4. 棋盘与地形设计

### 4.1 棋盘大小

不同阶段使用不同棋盘尺寸：

| 阶段 | 棋盘大小 | 说明 |
| --- | --- | --- |
| 教学 | 9x9 | 节奏快，便于理解规则 |
| 第一章 | 11x11 | MVP 推荐尺寸 |
| 中期章节 | 13x13 | 战略空间增加 |
| 后期章节 | 15x15 | 接近传统五子棋深度 |
| Boss | 特殊棋盘 | 可加入固定机关与地形 |

MVP 阶段建议使用 11x11。

### 4.2 MVP 地形

#### 普通格

无特殊效果，允许落子。

#### 灵脉格

效果：

- 在灵脉格落子时，获得 1 点能量。

设计目的：

- 制造资源争夺。
- 引导玩家和敌人在特定区域交锋。

#### 岩石格

效果：

- 无法落子。
- 阻断普通连线。

设计目的：

- 改变棋盘形态。
- 制造防守与绕路空间。
- 支持岩棋师与地形技能构筑。

### 4.3 后续扩展地形

#### 裂隙格

棋子落在此处后，若干回合后消失。适合快攻流。

#### 迷雾格

棋子落入后暂时隐藏归属。适合幻棋师与心理战。

#### 传送格

两个传送格在连线判定中视作相邻。适合星棋师与高阶玩法。

#### 荆棘格

落子需要额外消耗能量。适合资源压制关卡。

#### 圣坛格

在圣坛格上形成三连或四连时触发奖励。适合 Boss 机制。

## 5. 角色设计

### 5.1 MVP 角色

MVP 阶段建议只实现两个角色。

#### 剑棋师

定位：快攻连击型。

初始技能：破阵。

效果：

- 在己方二连或三连的一端落子时获得额外收益。
- 若本回合形成三连，额外获得 1 点能量。

被动：连势。

- 每当形成新的三连，下一次技能消耗降低 1。

设计目的：

- 鼓励主动进攻。
- 帮助玩家理解三连、四连、双线威胁。

#### 岩棋师

定位：防守控制型。

初始技能：立岩。

效果：

- 在一个空格生成岩石，持续 4 回合。

被动：固守。

- 成功阻止敌方四连时，获得 2 点能量。

设计目的：

- 展示地形系统价值。
- 提供稳健防守路线。

### 5.2 后续扩展角色

#### 幻棋师

核心机制：隐藏棋、假棋、迷雾格。

#### 星棋师

核心机制：传送门、跨空间连线、同调。

#### 血棋师

核心机制：牺牲己方棋子换资源与爆发。

## 6. 技能系统

### 6.1 技能设计原则

技能应满足以下原则：

1. 不能无成本直接赢得比赛。
2. 应制造策略选择，而不是直接给出答案。
3. 使用前必须有清晰预览。
4. 必须能被 AI 基本理解。
5. 尽量通过数据配置扩展。

### 6.2 MVP 技能列表

#### 破阵

类型：进攻强化。  
费用：2。  
效果：本回合在己方二连或三连一端落子时，如果形成三连，获得 1 点能量。

#### 双生子

类型：生成临时棋。  
费用：3。  
效果：本回合落子后，在相邻空格生成一颗临时棋。临时棋 2 回合后消失，且不能直接完成最终连五。

#### 立岩

类型：地形控制。  
费用：2。  
效果：在一个空格生成岩石，持续 4 回合。

#### 碎岩

类型：地形破坏。  
费用：2。  
效果：移除一个岩石格。

#### 封手

类型：防守反制。  
费用：3。  
效果：选择一个空格，敌方下回合不能在此落子。

#### 预警

类型：辅助提示。  
费用：1。  
效果：显示敌方下回合最危险的落点或威胁线。

### 6.3 技能数据结构建议

技能应数据化配置，例如：

```json
{
  "id": "rock_create",
  "name": "立岩",
  "cost": 2,
  "target": "empty_cell",
  "timing": "instant",
  "rarity": "common",
  "tags": ["terrain", "defense"],
  "effect": {
    "type": "create_terrain",
    "terrain": "rock",
    "duration": 4
  }
}
```

### 6.4 技能标签

建议使用标签支持构筑和奖励：

- attack
- defense
- terrain
- fog
- teleport
- summon
- temporary
- control
- resource

示例被动：

- 所有 terrain 技能费用 -1。
- 每次使用 summon 技能获得 1 点能量。
- fog 技能持续时间 +1。

## 7. 特殊棋子设计

### 7.1 普通棋

默认棋子，参与连线判定。

### 7.2 临时棋

效果：

- 若干回合后消失。
- 默认不能作为最终连五的第五颗棋。

用途：

- 快速制造威胁。
- 临时阻挡敌人。
- 配合快攻流。

### 7.3 强化棋

效果：

- 不能被推开。
- 不能被封印。
- 不受部分移除效果影响。

### 7.4 影棋

后续扩展。用于迷雾和隐藏信息玩法。

### 7.5 爆裂棋

后续扩展。用于区域威慑和破坏阵型。

## 8. 敌人与 AI 设计

### 8.1 AI 总体策略

MVP 阶段使用启发式评分 AI，而不是复杂搜索。

每个可落点计算以下分数：

```text
总分 =
  立即获胜分
+ 阻止玩家获胜分
+ 形成四连分
+ 阻止玩家四连分
+ 形成三连分
+ 阻止玩家三连分
+ 占据灵脉分
+ 避开负面地形分
+ 配合技能分
+ 敌人性格权重修正
```

### 8.2 AI 权重示例

```json
{
  "enemy_id": "fast_attacker",
  "weights": {
    "win_now": 10000,
    "block_win": 9000,
    "make_four": 800,
    "block_four": 500,
    "make_three": 300,
    "block_three": 200,
    "resource_cell": 50,
    "attack_multiplier": 1.4,
    "defense_multiplier": 0.8
  }
}
```

### 8.3 MVP 敌人

#### 新手棋魂

特点：

- 会阻挡明显三连。
- 较少制造复杂陷阱。

用途：教学与第一场战斗。

#### 快攻棋士

特点：

- 偏向形成二连、三连。
- 更积极制造冲四。

克制方式：稳健防守，拖到中后期。

#### 堡垒棋士

特点：

- 偏向防守与堵点。
- 会利用岩石格。

克制方式：碎岩、绕线、双线威胁。

#### 岩王

第一章 Boss。

能力：

- 开局生成岩石阵。
- 每 4 回合生成一块岩石。
- 岩石附近的 Boss 棋子视为强化棋。

Boss 棋盘：

- 中央有圣坛或灵脉。
- 局部岩石形成防线。
- 玩家需要争夺关键点。

## 9. Roguelike 系统

### 9.1 路线结构

每次冒险由多个节点组成：

- 普通战斗。
- 精英战斗。
- 事件。
- 商店。
- 休息点。
- 宝箱。
- Boss。

MVP 路线示例：

```text
起点
  -> 普通战斗
    -> 普通战斗 / 事件
      -> 精英 / 商店
        -> 休息点
          -> Boss
```

### 9.2 奖励三选一

战斗胜利后提供三选一奖励：

- 主动技能。
- 被动能力。
- 特殊棋子。
- 地形道具。
- 能量上限提升。

### 9.3 稀有度

| 稀有度 | 内容示例 |
| --- | --- |
| 普通 | 小技能、一次性道具、少量金币 |
| 稀有 | 强化某类技能或棋子 |
| 史诗 | 改变战斗节奏的能力 |
| 传说 | 构筑核心能力 |

### 9.4 事件节点示例

#### 棋盘裂缝

选择：

1. 支付金币修复裂缝，获得地形道具。
2. 利用裂缝，下一场增加裂隙格但奖励提高。
3. 离开。

#### 沉默棋手

选择：

1. 接受残局挑战，胜利获得稀有技能。
2. 偷看棋谱，获得预警技能，但下一场敌人先手。
3. 拒绝。

## 10. 构筑流派

### 10.1 快攻连杀流

核心：快速形成三连、四连，用技能制造双线威胁。

关键组件：

- 剑棋师。
- 破阵。
- 双生子。
- 三连回能。
- 额外行动奖励。

### 10.2 岩壁防守流

核心：用岩石和封手控制棋盘，慢慢建立不可阻挡的阵型。

关键组件：

- 岩棋师。
- 立岩。
- 封手。
- 强化棋。
- 碎岩反击。

### 10.3 地脉资源流

核心：争夺灵脉格获得大量能量，用技能频率压制敌人。

关键组件：

- 灵脉格。
- 点脉。
- 能量上限提升。
- 高费技能。

### 10.4 后续流派

- 迷雾刺杀流。
- 星门空间流。
- 血祭爆发流。

## 13. 核心代码架构

### 13.1 逻辑与表现分离

项目应严格区分：

- 纯逻辑层：棋盘状态、规则检测、AI、技能结算。
- 表现层：场景节点、动画、音效、UI。

错误示例：

```text
点击格子节点 -> 格子节点直接修改棋盘数据 -> 格子节点判断胜负
```

推荐流程：

```text
玩家点击格子
  -> 输入层生成 Action 请求
  -> BattleController 校验行动
  -> BoardState 修改数据
  -> RuleChecker 检查胜负
  -> View 层根据状态刷新表现
```

### 13.2 Action 系统

所有战斗行为都应抽象成 Action。

建议类型：

- PlacePieceAction。
- UseSkillAction。
- CreateTerrainAction。
- RemoveTerrainAction。
- ApplyStatusAction。
- GainEnergyAction。
- MovePieceAction。

优势：

- 支持撤销。
- 支持回放。
- 支持技能预览。
- 支持 AI 模拟。
- 支持战斗日志。

### 13.3 BoardState

职责：

- 保存棋盘宽高。
- 保存格子地形。
- 保存棋子位置。
- 判断格子是否可落子。
- 处理棋子移动和移除。
- 提供状态查询接口。

建议接口：

```text
get_cell(x, y)
get_piece_at(x, y)
get_terrain_at(x, y)
is_cell_playable(x, y)
place_piece(x, y, owner)
remove_piece(x, y)
move_piece(from, to)
```

### 13.4 RuleChecker 与 LineDetector

职责：

- 检测五连。
- 检测三连和四连。
- 检测活三和冲四。
- 判断封印、临时棋、岩石等规则影响。

基础方向：

```text
横向:   (1, 0)
纵向:   (0, 1)
斜向:   (1, 1)
反斜向: (1, -1)
```

### 13.5 SkillExecutor

技能执行流程：

```text
选择技能
  -> 检查能量
  -> 检查目标是否合法
  -> 生成预览
  -> 玩家确认
  -> 应用效果
  -> 扣除能量
  -> 触发被动
  -> 刷新棋盘
```

### 13.6 PreviewResult

为支持技能预览，建议实现模拟接口：

```text
simulate_action(action) -> PreviewResult
```

PreviewResult 应包含：

- 受影响格子。
- 受影响棋子。
- 能量变化。
- 地形变化。
- 是否形成威胁。
- 是否形成胜利。
- 是否非法。
- 非法原因。

## 15. MVP 范围

### 15.1 MVP 目标

MVP 的目标不是内容丰富，而是验证核心玩法是否成立：

1. 加入技能后，五子棋是否更有趣？
2. 地形是否真的改变策略？
3. Roguelike 奖励是否能形成不同构筑？
4. AI 是否能在新机制下提供基础挑战？

### 15.2 MVP 必做内容

#### 战斗系统

- 11x11 棋盘。
- 玩家与 AI 轮流落子。
- 五连判断。
- 三连、四连检测。
- 胜负结算。

#### 地形系统

- 普通格。
- 灵脉格。
- 岩石格。

#### 技能系统

- 能量。
- 技能栏。
- 技能目标选择。
- 技能预览。
- 6 个 MVP 技能。

#### AI 系统

- 启发式评分 AI。
- 3 个普通敌人。
- 1 个 Boss。

#### Roguelike 系统

- 简单路线图。
- 战斗奖励三选一。
- 1 个事件节点。
- 1 个商店节点。
- 1 个休息点。
- 1 个 Boss 节点。

### 15.3 MVP 暂不做内容

- 联机 PvP。
- 复杂账号系统。
- 云存档。
- 内购。
- 排行榜。
- 大量剧情。
- 多章节。
- 传送门。
- 影棋。
- 血祭流。
- 移动端适配。

## 16. 开发阶段规划

### 阶段 1：纯五子棋核心

目标：完成可玩的基础战斗。

任务：

- 创建棋盘场景。
- 实现落子。
- 实现五连检测。
- 实现玩家与简单 AI 轮流行动。
- 实现胜负界面。

验收标准：

- 玩家可以完成一局普通五子棋。
- AI 可以合法落子。
- 五连胜负判断正确。

### 阶段 2：地形系统

目标：证明地形能改变策略。

任务：

- 添加灵脉格。
- 添加岩石格。
- 地形影响可落子判断。
- 灵脉格提供能量。
- 岩石阻断连线。

验收标准：

- 玩家和 AI 都能识别地形。
- 地形效果在 UI 上清晰可见。

### 阶段 3：技能系统

目标：形成“技能 + 五子棋”的核心体验。

任务：

- 添加能量系统。
- 添加技能栏。
- 实现技能目标选择。
- 实现技能预览。
- 完成 6 个 MVP 技能。

验收标准：

- 每个技能可正常使用。
- 技能不能非法直接破坏胜负规则。
- 技能预览清晰。

### 阶段 4：AI 与敌人个性

目标：让不同敌人有不同棋风。

任务：

- 实现评分 AI。
- 添加快攻权重。
- 添加防守权重。
- 添加资源争夺权重。
- 实现岩王 Boss 基础机制。

验收标准：

- 快攻敌人表现更主动。
- 防守敌人更重视堵点。
- Boss 有明显机制特色。

### 阶段 5：Roguelike 外层

目标：完成一轮 Run。

任务：

- 实现路线地图。
- 实现奖励三选一。
- 实现事件节点。
- 实现商店节点。
- 实现休息点。
- 实现 Boss 节点。
- 实现本地存档。

验收标准：

- 玩家可以从起点推进到 Boss。
- 胜利后获得奖励并影响后续战斗。
- 失败后能重新开始。

### 阶段 6：打磨与扩展

目标：提升可玩性与表现。

任务：

- 增加动画与音效。
- 增加更多奖励。
- 调整数值平衡。
- 添加新角色。
- 添加新地形。
- 优化教程。

## 17. 测试策略

### 17.1 规则测试

重点测试：

- 横向五连。
- 纵向五连。
- 两种斜向五连。
- 岩石阻断连线。
- 封印棋不参与连线。
- 临时棋消失。
- 临时棋是否能参与最终胜利。

### 17.2 技能测试

每个技能至少测试：

- 能量不足时不可用。
- 非法目标不可选。
- 合法目标可执行。
- 执行后状态正确。
- 预览结果与实际结果一致。

### 17.3 AI 测试

重点测试：

- AI 能立即获胜时应优先获胜。
- 玩家即将获胜时 AI 应优先阻止。
- AI 不应在岩石格落子。
- AI 应能争夺灵脉格。
- 不同敌人权重表现有差异。

### 17.4 Roguelike 测试

重点测试：

- 路线节点生成合法。
- 奖励池不会为空。
- 奖励能正确应用到后续战斗。
- 存档和读档正确。
- Boss 节点能正常触发。

## 18. 平衡原则

### 18.1 技能不能廉价获胜

限制建议：

- 额外落子不能直接完成连五。
- 临时棋默认不能作为最终胜利棋。
- 复制棋如果形成五连，需要下一回合才判定。
- 强控制技能需要较高能量费用。

### 18.2 流派之间应有克制关系

| 流派 | 克制 | 被克制 |
| --- | --- | --- |
| 快攻 | 慢启动敌人 | 岩壁防守 |
| 岩壁 | 快攻 | 碎岩、星门 |
| 资源 | 长局 | 快攻 |
| 迷雾 | 普通防守 | 侦察 |
| 星门 | 固定防线 | 能量压制 |
| 血祭 | 逆风局 | 稳定消耗 |

### 18.3 战斗时长目标

PC 版本建议：

- 普通战斗：5-10 分钟。
- 精英战斗：8-15 分钟。
- Boss 战斗：15-25 分钟。

移动端版本应缩短：

- 普通战斗：3-6 分钟。
- 精英战斗：5-10 分钟。
- Boss 战斗：8-15 分钟。

## 21. 风险与应对

### 21.1 风险：规则复杂导致玩家困惑

应对：

- 分阶段教学。
- 清晰提示。
- 新机制逐章解锁。
- 技能描述保持短句。

### 21.2 风险：AI 无法理解复杂机制

应对：

- MVP 只做少量地形和技能。
- AI 使用可调权重。
- 技能加入 AI 使用评分。
- 后续对 Boss 单独写策略。

### 21.3 风险：技能预览开发量较大

应对：

- 从第一版就建立 Action 和 simulate_action。
- 不要把技能效果写死在 UI 中。
- 预览和实际执行共用同一套规则。

### 21.4 风险：单局战斗过长

应对：

- 使用 9x9 或 11x11 棋盘。
- 增加限回合奖励。
- 增加强进攻技能。
- 让敌人更主动制造威胁。

## 24. 开发注意事项

- 不要一开始做联机。
- 不要过早扩展角色和章节。
- 不要让技能描述过长。
- 不要把规则写进 UI 节点。
- 不要让 AI 完全无视地形。
- 不要让技能直接产生不可反制胜利。
- 优先做可玩性，再做美术量。
- 优先验证战斗循环，再做完整 Roguelike 外层。

## 25. 总结

本项目推荐采用 **Godot 4.x + GDScript** 进行开发，先以 PC 单机版本为目标，完成一个 11x11 棋盘的 MVP。MVP 应聚焦于五子棋核心、地形变化、技能系统、基础 AI 和 Roguelike 奖励构筑，不应过早加入联机、移动端适配或过多复杂机制。

最重要的开发原则是：

> 五子棋规则是地基，技能和地形是变化，Roguelike 构筑是长期乐趣。

只要 MVP 能证明“每局都有不同策略，每次失败都想再来一次”，这个项目就具备继续扩展为完整游戏的价值。
